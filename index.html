Areas de GIT

Directorio de trabajo

Es la carpeta donde realizamos los cambios en nuestro proyecto, en este lugar rara vez utilizaremos la consola y es el area donde se visualizan todos los cambios de los archivos.

  • mkdir & lt; dir > crea un directorio con el nombre especificado. Por ejemplo para crear una carpeta deberiamos escribir mkdir proyecto-git
  • cd & lt; dir > cambia el directorio actual por el indicado. Para acceder a una carpeta, la ruta debe ser correcta como cd proyecto-git
  • rm & lt; dir > elimina el directorio o archivo indicado. Para eliminar un repositorio de git debemos utilizarlo de la siguiente manera rm -rf .git
  • git init inicia un repositorio de git o lo reinicia en caso que este ya existe. Debemos tener en cuenta que si utilizamos este comando las ubicaciones diferentes, cada ubicación se considera un repositorio en si.

Área de preparación (índice)

Es el area de control de cambios, donde se verifican las modificaciones realizadas en los archivos del proyecto. Aqui se realizan los instantáneas de codigo para luego ser confirmados.

  • git status verifica los cambios del DIRECTORIO DE TRABAJO y los compara con lo almacenado en el INDICE. Cada archivo posee un estado segun las acciones realizadas
    • untracked: no se esta haciendo seguimiento
    • seguido: se esta haciendo seguimiento
    • modificado: difiere con lo que se encuentra en el índice
    • comprometido: los cambios estan registrados
  • git add & lt; file > agrega al INDEX los archivos especificados, también se pueden agregar directorios, siempre y cuando estos tengan archivos en su interior. Un uso rápido de este comando es git add. que agrega todos los archivos que se hayan modificado o creado en el directorio actual
  • git rm --cached & lt; file > Quita el archivo del área de seguimiento, pero lo mantiene en el DIRECTORIO DE TRABAJO. Este comando se utiliza cuando no existe una referencia para reestablecer el contenido.
  • git restore & lt; file > Quita el archivo del INDEX, y lo reestablece a como estaba en la ultima cabecera.

Repositorio (local / remoto)

Es el lugar donde se almacen las capturas en formato BLOB (Archivo de objeto binario) a las cuales podemos acceder mediante el historial de cambios. Para generar estos archivos debemos emitir un commit, el mismo valida los cambios y los almacena en la base de datos de GIT.

  • git commit Abre el editor para generar una confirmación de cambios que será guardada en la base de datos del repositorio. Un uso rápido de este comando es git commit -m "message" que permite generar un commit con mensaje sin abrir el editor.
  • git diff & lt; ref > Compara el DIRECTORIO DE TRABAJO con el compromiso al cual apunta la referencia especificada. una forma de comparar cambios realizados cuando no se realizo un commit es git diff HEAD , pero si queremos saber como difiere nuestra cabecera con respecto a otro commit podemo indicarlo de diferentes maneras.
    • referencia absoluta: especificamente hablando es el numero de HASH del commit que queremos comparar, por ejemplo c215714
    • referencia relativa indica cuantos cometer atras esta nuestra referencia, por ejemplo HEAD ^^ indica dos ancestros antes de HEAD y HEAD ~ 4 indica 4 comete antes de HEAD
      • ul >
    • git blame & lt; file > muestra las lineas de codigo del archivo especificado, colocando al principio en numero de commit, el nombre del autor, la fecha y hora en que se realizo el mismo, e incluso indica cuando los cambios todavia no fueron confirmados.
    • git remote lista los repositorios remotos que hayan sido agregados, si nuestro proyecto es un clon por defecto se agrega un remoto llamado origin , pero si fue creado, debemos hacerlo manualmente usando el comando git remote add & lt; alias > & lt; URL >

    Apuntadores de GIT

    CABEZA

    Es un apuntador movil que nos permite navegar en el historial de cambios, normalmente apunta al ultimo commit de la rama donde nos encontremos y podemos cambiar su posicion para poder visualizar el proyecto a como se mana en algun punto de referencia (commit) utilizando el comando git checkout & lt; ref >

    SUCURSAL

    Las ramas son apuntadores dinamicos que siempre hacen referencia al ultimo commit dentro del historial. Podemos crear diferentes ramas y cada una apuntara a un commit particular, este puede ser el mismo o pueden ser totalmente diferentes. Si se quiere comparar, podrian interpretar como lineas de tiempo paralelas; las ramas son necesarias ya que permiten al apuntador HEAD navegar en el historial y volver al ultimo commit sin problemas.
